# -
读后感  
  作为一名之前对区块链和比特币毫无概念的纯小白，第一次去读Satoshi的原始论文，在翻开白皮书之前，我脑海里的“比特币”无非就是一种网上的虚拟货币。我当时想：像腾讯的Q币或者支付宝里的余额一样，在服务器的数据库里给我的账号后面加一串数字但我一读第二章“交易（Transactions）”，白皮书里写着：比特币这个系统里，根本没有“账户余额”这个概念。我原本以为，两人之间的交易，系统会把一人的余额减去，一人的余额加上。但白皮书里展示的却是一条由“电子签名”连接起来的交易链。它不是记录“你有多少钱”，而是记录“这笔钱从哪里来，要到哪里去”。我一开始想错了，以为有了非对称加密和电子签名，转账就是绝对安全的。但读了第二遍我才明白：电子签名只能防伪造，却防不了“双花”——即我把同一笔钱，同时转给两个人。若不想引入类似于银行功能的“铸币厂”，但更好的实现点对点的交易，我们需要一种方式，让收款方知道之前的拥有者从未签署过任何早期交易，针对此白皮书引入了工作量证明，第四章和第五章的每一个概念都需要额外去查询资料理解。首先我去深入查了什么是哈希函数（SHA-256）。白皮书里写道：节点需要寻找一个随机数（Nonce），使得区块的哈希值（Hash）满足一定数量的前导零。为什么哈希值一定要有零？为什么非要让全世界的电脑去拼命计算这个毫无实际意义的“随机数”？在去理解哈希值解算过程之后我才明白，其实前导零的设置相当于只是设置一个“中将条件”，零的个数，就是控制这个“中奖概率”的阀门，这其实是一个“用物理世界的客观规律，去锁死虚拟世界秩序”的精妙设计。因为哈希计算是不可逆的，想要找到那个符合要求的随机数，没有任何捷径，只能像买彩票一样一个一个去试。这就意味着，谁投入的计算资源多，谁就掌握了话语权（最长链）。Satoshi 用这种听起来极其粗暴、甚至有些“大智若愚”的暴力计算，硬生生地在没有任何监管、谁也不认识谁的网络里，建立起了一个“铁律”：只要大部分人的算力是诚实的，历史就永远无法被篡改。这种用算力代替信用、用数学代替人性的设计。  
  除了硬核的数学逻辑，白皮书里最让我反复去思考的，是第五章对“网络运行（Network）”的设计。在我的常识里，银行系统、支付系统的网络要求必须是高精度、绝对完美的。哪怕一毫秒的延迟、一个数据包的丢失，都可能导致账目对不上。因此，这些系统需要极其昂贵的服务器和绝对稳定的专线。但比特币白皮书里对网络的要求并不需要做到实时同步，论文中明确写道，如果有交易或区块在广播时丢失了（也就是说某些节点没收到），根本不要紧。只要后面有节点收到了新区块，它们就会自动意识到自己漏掉了什么，并在需要的时候主动去请求下载。节点想来就来，想走就走，只要连上后直接去要“最长的链”就行，最长链就是一切的历史真相。同时还有在第六章“激励（Incentive）”中，Satoshi讨论了如果有人掌握了超过50%的算力，系统该怎么办。我原以为这会是一个复杂的防范算法，但白皮书里的解决方案居然是“搞好关系、利诱坏人”。他写道，如果一个攻击者拥有了这么强大的算力，他有两个选择：搞破坏：撤销自己的交易，把钱拿回来，从而毁掉整个比特币系统的信用。顺从规则：用这些算力去挖矿，赚取系统奖励的新比特币。Satoshi极其自信地指出，第二种选择能让攻击者赚到更多的钱。因为一旦他把系统搞垮了，他手里抢到的比特币也就变成了废纸。用自私的贪婪来维护系统的安全，这不仅仅是一种死板的设计更是对人性的一个考量。  
  作为初学者，白皮书里仍然有大半内容对我来说不太容易理解。尤其是以下两个部分：Satoshi为了证明“诚实节点战胜攻击者的概率”，给出的数学推导我不太能理解。以及第七章提到了为了节省空间，可以把旧区块里的交易用默克尔树（Merkle Tree）的形式折叠起来，只保留一个根哈希（Root Hash）。白皮书里画了一个层层向上的树状图。我不太理解为什么把底下的交易删掉、只留一个哈希值，节点依然能验证这笔交易的真实性？难道信息不会丢失吗？这种在数学上实现“无损压缩且可验证”的操作目前对我来说无法太理解。
